# 네잎클로버 찾기 — 프로젝트 작업 가이드

> 이 디렉토리에서 작업할 때 Claude가 먼저 읽어야 할 문서.
> 게임 자체의 컨셉·디자인·로드맵은 [PROJECT.md](PROJECT.md)에 있음 — 항상 함께 참조.

---

## 디렉토리의 정체

이 디렉토리는 **"네잎클로버 찾기"라는 모바일 힐링 게임 프로젝트의 모든 것**을 담는 공간이다.
게임 코드뿐 아니라 마케팅 자료, 리서치 노트, 디자인 레퍼런스, 비즈니스 메모 등 게임 출시까지 필요한 모든 산출물이 여기 모인다.

```
four-leaf-clover/
├── client/          ← 게임 코드 (Phaser.js + Capacitor + TypeScript)
├── PROJECT.md       ← 게임 디자인·로드맵 문서 (단일 출처)
├── CLAUDE.md        ← 이 파일
└── (추후 추가)      ← 마케팅/리서치/디자인 자료 폴더 — 구조는 자료가 쌓이면서 결정
```

- 게임 코드 관련 작업은 **`client/` 안에서** 진행한다. 루트에 소스 파일을 두지 않는다.
- 마케팅/리서치/디자인 자료의 폴더 구조는 아직 미정이다. 새 자료를 추가해야 할 일이 생기면 **먼저 사용자와 폴더명/위치를 합의**한 뒤 만든다.

## 작업 컨텍스트 (항상 인지)

- **현재 Phase**: Phase 1 — 코어 루프 검증 단계. *"찾는 행위가 재밌는가?"* 에 답하는 게 최우선.
- **기술 스택**: Phaser.js + TypeScript + Vite + Capacitor. 자세한 근거는 PROJECT.md §4.
- **아트**: Phase 1은 코드 생성 SVG 도형으로 충분. 손그림 에셋은 Phase 2부터.

---

## 코딩 시 지켜야 할 것 (PROJECT.md §7 요약 + 보강)

- **TypeScript 우선**. 게임 오브젝트 타입은 명시적으로.
- **Phase 1에서는 과도한 추상화 금지**. 빠르게 만들고 빠르게 검증. 가설을 검증하기 전에 인터페이스부터 일반화하지 않는다.
- **상태 관리**: Phase 1~2는 Phaser 내장 상태로 충분. Capacitor Preferences / SQLite는 Phase 3부터 고려.
- **테스트**: `FieldGenerator` 같은 순수 로직만 단위 테스트. UI/연출은 수동 검증.
- **패키지 매니저**: `client/`는 **pnpm**을 쓴다 (`packageManager` 필드로 고정). `npm install` 금지. 새 의존성 추가는 `pnpm add <pkg>`.
  - Capacitor가 iOS 네이티브를 추가하는 Phase 4 시점에 CocoaPods가 심볼릭 `node_modules`를 못 따라가는 이슈가 생기면, `client/.npmrc`에 `node-linker=hoisted` 한 줄 추가로 해결.

## Git 사용 규칙

- **저장소 범위**: 루트(`four-leaf-clover/`) 전체가 하나의 git 저장소. 코드(`client/`)와 마케팅·리서치 자료를 한 저장소에서 함께 추적한다.
- **커밋 시점 — 주요 단계마다 반드시 커밋을 남긴다**:
  - 코드: PROJECT.md §6의 작업 단위(6.1~6.5)가 자연스러운 커밋 경계. *각 작업이 끝나면* 커밋.
  - 마케팅/리서치/디자인 자료: 의미 있는 자료 추가·갱신이 끝날 때마다 커밋. "초안 - 진행 중"을 길게 끌지 않는다.
  - 의존성·설정 변경(`package.json`, `vite.config`, `capacitor.config` 등)은 코드 변경과 분리해서 별도 커밋.
- **커밋 메시지**: 한국어로 작성. 첫 줄은 60자 이내 요약, 필요 시 본문에 *왜* 그렇게 했는지. 예: `Clover 엔티티 — leaves/hue/rotation 파라미터 추가`.
- **커밋 단위**: 한 커밋 = 하나의 논리적 변경. 무관한 변경을 묶지 않는다. 진행 중인 상태로 커밋하지 않는다(빌드·타입체크는 통과해야 함).
- **명시적 요청 전에는 커밋하지 않는다**: 사용자가 "커밋해" 또는 그에 준하는 지시를 줬을 때만 실제 `git commit`을 실행한다. 그 외에는 *"이 시점에서 커밋하는 게 좋겠다"* 정도로 제안만.
- **푸시·force push 등 원격 작업**: 항상 사용자 확인 후 실행.

## 사용자와 일하는 방식

- **언어**: 기본 한국어로 응답. 코드·식별자·기술 용어는 영어 그대로.
- **결정 시점이 명시된 항목 (PROJECT.md §2.4)** 은 아직 결정하지 않는다. 사용자가 그 시점에 도달했다고 신호를 주기 전까지는 *"이건 Phase N에서 결정"* 으로 보류한다.
- **새 폴더·새 파일 생성**: 코드 파일은 `client/` 구조 안에서 자유롭게. 루트 디렉토리에 새 폴더(예: `marketing/`, `docs/`)를 만들기 전에는 사용자에게 확인.
